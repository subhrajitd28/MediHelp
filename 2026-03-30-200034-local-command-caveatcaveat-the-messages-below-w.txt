     MediHelp - Complete Microservice Architecture Plan

     Context

     Problem: People need guidance before seeing a doctor and structured
     follow-up after appointments, but existing apps are either too generic or
     too expensive.

     MediHelp is a personalised health assistant that provides pre-diagnosis
     support (symptom triage, self-assessment), post-appointment monitoring
     (medication tracking, follow-ups), and continuous health management
     (vitals, diet, exercise) -- all built as a microservice architecture using
     free student resources.

     Team: 3-4 college students | Backend: Spring Boot 3.x | Frontend: Angular
     17+ | AI Service: Python FastAPI

     ---
     1. Unique Differentiating Features (What Makes This Stand Out)

     1.1 AI-Powered Symptom Triage (Gemini API - free tier)

     - Users describe symptoms in plain language
     - AI classifies urgency (low/medium/high/emergency), suggests potential
     conditions (with disclaimers), recommends next steps
     - Tracks symptom history over time to detect worsening patterns
     - Built-in rate-limit queue with backoff to handle Gemini free tier limits
     (~1,000 req/day)

     1.2 Prescription OCR + Drug Interaction Checker

     - Stage 1: Tesseract 5.x OCR with OpenCV preprocessing (grayscale, adaptive
      thresholding, deskewing)
     - Stage 2: User confirms/corrects extracted text (realistic ~80-85%
     accuracy)
     - Stage 3: Drug interaction checking via OpenFDA Drug Label API + RxNorm
     API (both free, government-backed)
     - Auto-creates medication reminders from scanned prescriptions

     1.3 FHIR-Compliant Health Data (Industry Standard)

     - Internal data models map to FHIR R4 resources: Patient, Observation,
     MedicationRequest, Condition, AllergyIntolerance
     - /fhir/export endpoint serializes user data as a FHIR Bundle (JSON)
     - Uses HAPI FHIR library (open source Java) for serialization
     - This is a genuine differentiator -- demonstrates healthcare
     interoperability awareness

     1.4 Anomaly Detection in Vitals (No ML needed)

     - Threshold-based: Configurable per user (e.g., systolic BP > 140, blood
     sugar > 200 mg/dL)
     - Trend-based: Moving average over 7-14 readings + 2-standard-deviation
     check using Welford's algorithm (incremental, no re-querying history)
     - Immediate alerts on anomalies via push/email

     1.5 Family Health Hub

     - Users create "dependents" (elderly parents, children) with roles: OWNER,
     CAREGIVER, VIEWER
     - Unified dashboard showing all family members' health status
     - Each dependent has own vitals, medications, records, appointments

     1.6 Gamified Health Score (0-100)

     - Medication adherence (25pts) + Vitals consistency (20pts) + Exercise
     goals (20pts) + Diet adherence (15pts) + Appointment attendance (10pts) +
     Mood journal (10pts)
     - Streaks: Consecutive days of full adherence with flame icon, streak
     freeze once/week
     - Badges: "First Week Warrior", "30-Day Streak", "Vitals Veteran", "Family
     Champion" -- stored in config table, not hardcoded

     1.7 Emergency SOS

     - One-tap: captures GPS, compiles medical summary (allergies, meds, blood
     type, recent vitals)
     - Sends to pre-configured emergency contacts via email
     - Generates time-limited shareable link (24h expiry, cryptographically
     random URL) for first responders

     1.8 Mental Health Mood Journaling

     - Daily 5-point mood scale + optional encrypted journal entry
     - Mood trends graph + correlation overlay with medications, exercise, sleep

     2.3 Communication Patterns

     Synchronous (REST via Gateway):
     - All client requests route through Gateway
     - Inter-service calls use Spring Cloud OpenFeign with Eureka discovery
     - Circuit breakers via Resilience4j on all Feign clients

     Asynchronous (RabbitMQ via CloudAMQP free tier):
     Exchange: medihelp.events (topic exchange)

       user.registered           -> User Profile Svc (create default profile)
                                 -> Notification Svc (send welcome email)
       medication.reminder.due   -> Notification Svc (send push/email)
       vitals.anomaly.detected   -> Notification Svc (send urgent alert)
       appointment.reminder.due  -> Notification Svc (send reminder)
       health.score.updated      -> Notification Svc (badge earned notification)
       medication.adherence.updated -> Health Tracking Svc (recalculate score)

     Scheduled Tasks (@Scheduled):
     - Prescription Service: every minute, check upcoming medication reminders
     - Health Tracking: daily at midnight, recalculate health scores, check
     streaks
     - Notification Service: every 30s, process notification queue + retry
     failed emails

     [Grafana Cloud Dashboards + Alerting]

     Dashboards to Build

     1. Service Health: UP/DOWN status, JVM memory, CPU, GC pauses, active

     2. API Performance: Request rate, error rate (4xx/5xx), latency percentiles

     4. Notification Delivery: Emails sent/failed, push delivered, reminder

     Alerting Rules

     - Service DOWN > 2 min -> email
     - Error rate > 5% for > 5 min -> email
     - DB approaching 0.5GB (> 400MB) -> email
     - RabbitMQ queue depth > 100 for > 10 min -> email

     8. Security

     - JWT: Access token (15min) + Refresh token (60 days, stored hashed in DB)
     - Gateway-level validation: JWT verified at gateway, X-User-Id +
     X-User-Roles forwarded to downstream services
     - Token blacklisting: On logout, JWT ID stored in Redis with TTL =
     remaining token lifetime
     - Rate limiting: Gateway-level via Redis -- 100 req/min unauthenticated,
     300 authenticated, 10 for auth endpoints, 20 for AI endpoints
     - Input validation: Jakarta Bean Validation on all DTOs, OWASP HTML
     Sanitizer for text fields
     - Encryption: Mood journal text AES-256-GCM encrypted at rest, passwords
     bcrypt(12), OTPs stored hashed
     - CORS: Configured only at Gateway level, allow frontend origin +
     localhost:4200
     - File uploads: Max 5MB, JPEG/PNG/PDF only
     - OTP flow: 6-digit, 5-min expiry, email-based via Resend (3,000/month is
     enough)

     9. Key API Endpoints

     Auth (/api/v1/auth): POST /register, /verify-otp, /login, /refresh,
     /logout, /forgot-password, /reset-password, GET /oauth2/google

     User Profile (/api/v1/users): GET|PUT /me, GET|POST /me/family/members,
     GET|POST /me/emergency-contacts, /me/allergies, /me/conditions, GET 
     /me/fhir/patient

     Health (/api/v1/health): POST|GET /vitals, GET /vitals/latest,
     /vitals/trends, /vitals/anomalies, GET /score, /score/history, GET 
     /streaks, /badges, POST|GET /mood, POST|GET /records, GET /fhir/bundle

     Prescriptions (/api/v1/prescriptions): POST|GET /, POST /{id}/ocr, POST|GET
      /medications, POST /medications/check-interactions, GET 
     /medications/schedule, POST /medications/log, POST|GET /appointments

     AI (/api/v1/ai): POST /symptoms/triage, /ocr/process,
     /diet/recommendations, /exercise/recommendations

     Notifications (/api/v1/notifications): GET /, PATCH /{id}/read, GET 
     /unread-count, GET|PUT /preferences

     All endpoints use SpringDoc OpenAPI for auto-generated Swagger docs at
     /swagger-ui.html.
